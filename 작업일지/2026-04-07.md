## 1. [보안] 평문 비교 폴백 제거

### 기존 상태
```java
if (adminPassword.startsWith("$2a$") || adminPassword.startsWith("$2b$")) {
    return ENCODER.matches(password, adminPassword);
}
return adminPassword.equals(password); // 평문 비교 폴백
```

### 문제점
환경변수에 평문 비밀번호를 그대로 넣으면 BCrypt 검증을 우회하여 평문 비교로 동작, BCrypt 도입이 무의미해짐.

### 변경 내용
```java
if (!(adminPassword.startsWith("$2a$") || adminPassword.startsWith("$2b$"))) {
    log.error("[SECURITY] admin.password가 BCrypt 해시가 아닙니다. 로그인이 차단됩니다.");
    return false;
}
return ENCODER.matches(password, adminPassword);
```
BCrypt 형식이 아닌 비밀번호로는 로그인 자체가 차단됨. `@PostConstruct`에서 기본값(`123456789`) 사용 시 ERROR 로그 출력.

## 2. [백엔드] GlobalExceptionHandler 생성

### 기존 상태
각 컨트롤러마다 예외 처리 없음 → Spring 기본 에러 페이지(HTML) 또는 500 반환.

### 문제점
API 오류 시 HTML 응답이 섞여 프론트에서 파싱 불가, 에러 원인 알 수 없음.

### 변경 내용
`common/exception/GlobalExceptionHandler.java` 신규 생성.
```java
@RestControllerAdvice
public class GlobalExceptionHandler {
    @ExceptionHandler(NoSuchElementException.class) → 404 { code, message }
    @ExceptionHandler(MethodArgumentNotValidException.class) → 400 { code, message }
    @ExceptionHandler(IllegalArgumentException.class) → 400 { code, message }
    @ExceptionHandler(Exception.class) → 500 { code, message }
}
```

## 7. [프론트엔드] fetch 타임아웃 설정

### 기존 상태
```js
const res = await fetch(url, finalOptions);
```

### 문제점
AI 서버 응답 지연 시 로딩 스피너가 영원히 돌고, 사용자는 강제 새로고침 외에 벗어날 방법 없음.

### 변경 내용
```js
const finalOptions = {
  credentials: 'include',
  signal: AbortSignal.timeout(10000),
  ...options,
};
```
모든 API 요청에 10초 타임아웃 자동 적용.

## 9. [프론트엔드] ProtectedRoute + UserProtectedRoute 구현

### 기존 상태
관리자/일반 사용자 페이지 모두 일반 `<Route>`로 노출.

### 문제점

### 변경 내용
- `ProtectedRoute.jsx`: `/api/admin/check` 호출 → 401이면 `/admin/login` 리다이렉트
- `UserProtectedRoute.jsx`: `/api/user/check` 호출 → 401이면 `/login` 리다이렉트
- `UserLoginPage.jsx`: 일반 사용자 비밀번호 로그인 페이지 (`/login` 라우트)
- `App.js`: 관리자 라우트 → `ProtectedRoute`, 일반 라우트 → `UserProtectedRoute`로 일괄 감쌈

## 11. [백엔드] API 응답형식 통일

### 기존 상태
각 컨트롤러마다 `List`, `Map`, DTO, `ResponseEntity` 등 응답 형식이 제각각.

### 문제점
프론트에서 API별로 다르게 파싱해야 하고, 에러 응답 구조도 일관성 없음.

### 변경 내용
**백엔드 신규:** `common/ApiResponse.java`
```java
public class ApiResponse<T> {
    private final boolean success;
    private final T data;
    private final String message;

    public static <T> ApiResponse<T> ok(T data) { ... }
    public static ApiResponse<Void> ok() { ... }
}
```

**래핑된 컨트롤러:** SettingsController, NotificationController, LogController, AlertController, DashboardController, ScanController (6개)

**프론트 자동 unwrap:** `HttpClient.js`에서 `{ success, data }` 구조 감지 시 `data` 자동 추출 → 기존 프론트 코드 변경 없음.

```js
if (json !== null && typeof json === 'object' && 'success' in json && 'data' in json) {
  return json.data;
}
```
