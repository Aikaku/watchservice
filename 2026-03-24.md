## 5. Electron 데스크탑 앱 배포 — Phase 1 + Phase 2 구현

### 기존 상태
React(포트 3000)와 Spring Boot(포트 8080)가 별도로 실행되는 구조였다. 배포 방식은 터미널 수동 실행이었고 Electron 관련 코드가 전혀 없었다.

### 문제점
- 포트가 2개라 API 요청 시 CORS 처리 필요
- 배포 시 사용자가 Java, Node.js를 별도 설치해야 했음
- 일반 사용자가 터미널 명령어로 실행해야 해서 진입 장벽이 높았음

### 변경 내용

**Phase 1 — React + Spring Boot 단일 포트 통합:**

`HttpClient.js`의 `BASE_URL`을 상대경로로 변경:
```js
// 변경 전
const BASE_URL = process.env.REACT_APP_API_BASE_URL || 'http://localhost:8080';

// 변경 후 (같은 포트에서 서빙되므로 상대경로 사용)
const BASE_URL = process.env.REACT_APP_API_BASE_URL || '';
```

`WebConfig.java`에 SPA 라우팅 처리 추가:
- `/api/**`, `/actuator/**` 외 모든 경로 → `index.html` 포워딩
- 정적 리소스(`*.js`, `*.css`, `*.ico` 등) 직접 서빙

통합 빌드 스크립트 작성:
- `build.sh` (macOS/Linux): React 빌드 → `resources/static/` 복사 → Gradle 빌드
- `build.bat` (Windows): 동일 과정 Windows 배치 파일로 구현

**Phase 2 — Electron Shell 구성:**

`watchservice_electron/` 폴더 신규 생성. 구성 파일:
- `package.json`: electron-builder 설정 (Windows `.msi` / macOS `.dmg`)
- `main.js`: Spring Boot JAR를 `child_process.spawn()`으로 실행, 헬스체크 폴링(500ms 간격, 60초 타임아웃), 시스템 트레이 등록, 창 닫기 → 트레이 최소화, 앱 종료 시 Spring Boot SIGTERM
- `preload.js`: contextBridge로 `platform` 정보 노출
- `assets/loading.html`: 시작 로딩 화면 (스피너)
- `assets/error.html`: Spring Boot 시작 실패 시 오류 안내 화면

**변경 파일 목록**:

| 파일 | 변경 내용 |

## 6. Phase 3 — AI 서버 클라우드 배포 준비

### 기존 상태
`application.yml`의 AI 서버 URL이 `http://localhost:8001/api/analyze`, `http://localhost:8001/predict`로 하드코딩되어 있었다. AI 서버 배포에 필요한 `requirements.txt`, `Procfile`, `railway.toml` 등이 존재하지 않았다.

### 문제점
- AI 서버 주소가 코드에 고정되어 있어 클라우드 배포 후 Spring Boot를 재빌드하지 않으면 URL을 바꿀 수 없었다.
- AI 서버를 Railway/Render에 올리려면 별도 배포 설정 파일이 필요하나 준비되지 않은 상태였다.

### 변경 내용

**`application.yml` — AI 서버 URL 환경변수화:**
```yaml
# 변경 전
ai:
  analyze:
    url: http://localhost:8001/api/analyze
  family:
    url: http://localhost:8001/predict

# 변경 후
ai:
  analyze:
    url: ${AI_ANALYZE_URL:http://localhost:8001/api/analyze}
  family:
    url: ${AI_FAMILY_URL:http://localhost:8001/predict}
```
- 기본값을 기존 localhost로 유지하여 개발 환경에서 환경변수 없이도 동작
- 배포 환경에서는 `AI_ANALYZE_URL`, `AI_FAMILY_URL` 환경변수로 클라우드 주소 주입

**`ai_server_deploy/` — AI 서버 배포 패키지 신규 생성:**
- `requirements.txt`: fastapi, uvicorn, numpy, pandas, xgboost, lightgbm, pyyaml, pydantic 버전 고정
- `Procfile`: `uvicorn api_server:app --host 0.0.0.0 --port $PORT`
- `railway.toml`: healthcheckPath=/health, restartPolicy on_failure
- `.env.example`: CORS_ALLOWED_ORIGINS 환경변수 설정 예시
- `configs/config.yaml`: 모델 파일 경로 및 서버 설정 템플릿
- `README_DEPLOY.md`: Railway 배포 단계별 절차 (레포 분리 → 배포 → Spring Boot 환경변수 주입 → 검증)

**변경 파일 목록**:

| 파일 | 변경 내용 |
|------|----------|
| `watchservice_be/src/main/resources/application.yml` | AI 서버 URL 환경변수화 (`AI_ANALYZE_URL`, `AI_FAMILY_URL`) |
| `ai_server_deploy/requirements.txt` | 신규 생성 — Python 패키지 의존성 |
| `ai_server_deploy/Procfile` | 신규 생성 — Railway/Render 시작 명령 |

| `ai_server_deploy/.env.example` | 신규 생성 — 환경변수 예시 |
| `ai_server_deploy/configs/config.yaml` | 신규 생성 — 모델 경로 설정 템플릿 |

## 7. Phase 4 — JRE 번들링 및 인스톨러 빌드

### 기존 상태
Electron 인스톨러 빌드 스크립트(`build.sh`, `build.bat`)는 JAR까지만 빌드하고 종료되었다. JRE 번들 생성 스크립트가 없었고, `package.json`의 `files` 배열에 `jre/**`, `app.jar`가 중복 포함되어 있었다. `main.js`에서 아이콘 파일이 없으면 앱이 크래시되는 문제와 시작 프로그램 등록 기능이 없었다.

### 문제점
- Java가 설치되지 않은 사용자 PC에서 앱이 실행되지 않음 (JRE 번들 없음)
- `files` + `extraResources` 중복으로 인스톨러 용량 불필요하게 증가
- 아이콘 파일 없을 때 `nativeImage.createFromPath()` 오류로 앱 시작 실패 가능
- 앱 설치 후 자동 시작 미지원 — 사용자가 매번 수동 실행 필요
- 트레이 메뉴에 시작 프로그램 등록/해제 옵션 없음

### 변경 내용

**`jre_build.sh` / `jre_build.bat` 신규 생성:**
- `jlink`로 Spring Boot 3.x 구동에 필요한 최소 JRE 생성
- 포함 모듈: `java.base`, `java.sql`, `java.net.http`, `jdk.unsupported`, `jdk.crypto.ec` 등 22개
- `--strip-debug`, `--no-man-pages`, `--no-header-files`, `--compress=2` 옵션으로 용량 최소화
- 출력 경로: `watchservice_electron/jre/`
- JAVA_HOME 미설정 시 `which java` / `where java`로 자동 탐지

**`build.sh` / `build.bat` 업데이트:**
```
기존: [1/3] React 빌드 → [2/3] 복사 → [3/3] Gradle 빌드
변경: 기존 3단계 유지 + --electron 플래그 시 [4/5] JRE 번들 → [5/5] Electron 인스톨러
```
- `./build.sh --electron` (macOS) → `.dmg` 생성
- `build.bat --electron` (Windows) → `.msi` 생성

**`package.json` 수정:**
- `files` 배열에서 `jre/**`, `app.jar` 제거 (extraResources와 중복이었음)
- `mac.minimumSystemVersion: "11.0"` 추가
- `win.requestedExecutionLevel: "asInvoker"` (UAC 없이 사용자 권한으로 실행)
- `dmg.contents` 배열 추가 (Applications 폴더 링크 포함)

**`main.js` 수정:**
- `loadIcon(filename)` 함수 추출 — 파일 없으면 `nativeImage.createEmpty()` 반환 (크래시 방지)
- 패키징된 앱 첫 실행 시 `app.setLoginItemSettings({ openAtLogin: true })` 자동 호출
- 트레이 메뉴에 "시작 프로그램 등록/해제" 토글 항목 추가 (`buildContextMenu()` 함수로 동적 생성)

**변경 파일 목록**:

| 파일 | 변경 내용 |
|------|----------|
| `jre_build.sh` | 신규 생성 — macOS/Linux jlink 최소 JRE 생성 스크립트 |
| `jre_build.bat` | 신규 생성 — Windows jlink 최소 JRE 생성 스크립트 |
| `build.sh` | `--electron` 플래그 추가, JRE + Electron 빌드 단계 추가 |
| `build.bat` | `--electron` 플래그 추가, JRE + Electron 빌드 단계 추가 |
| `watchservice_electron/package.json` | files 배열 정리, Windows/macOS 세부 설정 추가 |
| `watchservice_electron/main.js` | 아이콘 graceful 처리, 시작 프로그램 자동 등록, 트레이 토글 |

## 8. Phase 5 — 배포 사이트 구성

### 기존 상태
인스톨러를 빌드해도 사용자가 다운로드할 수 있는 웹 페이지가 없었다. GitHub Releases 생성도 수동으로 해야 했고, 자동화 워크플로가 없었다.

### 문제점
- 새 버전 릴리스 시마다 수동으로 macOS + Windows 두 환경에서 빌드 후 GitHub에 직접 업로드해야 했다.
- 사용자에게 공유할 다운로드 URL이 없었다.
- Windows/macOS 설치 방법에 대한 안내 문서가 없었다.

### 변경 내용

**`.github/workflows/release.yml` — 자동 릴리스 워크플로:**
- 트리거: `v*.*.*` 형식의 태그 푸시 (예: `git tag v1.0.0 && git push origin v1.0.0`)
- `build-mac` 잡: `macos-latest` 러너에서 React → JAR → JRE → `.dmg` 생성
- `build-win` 잡: `windows-latest` 러너에서 동일 과정으로 `.msi` 생성
- `release` 잡: 두 아티팩트를 수집해 GitHub Release 자동 생성 및 인스톨러 첨부

**`.github/workflows/pages.yml` — GitHub Pages 배포 워크플로:**
- `download_site/` 폴더 변경 시 자동으로 GitHub Pages에 배포
- 완료 기준: `https://GITHUB_USERNAME.github.io/REPO_NAME/`에서 다운로드 페이지 접근 가능

**`download_site/index.html` — 다운로드 랜딩 페이지:**
- OS 자동 감지 (`navigator.userAgent`) → 해당 OS 카드 강조 표시
- Windows(.msi) / macOS(.dmg) 다운로드 버튼 (GitHub Releases latest 링크)
- 설치 단계 4단계 안내 (다운로드 → 설치 → 실행 → 감시폴더 설정)
- macOS 보안 경고 안내
- 다크 테마, 반응형 레이아웃

**`download_site/INSTALL_GUIDE.md` — 설치 가이드:**
- Windows: MSI 설치, SmartScreen 경고 처리, 제거 방법
- macOS: DMG 설치, "확인되지 않은 개발자" 경고 처리 2가지 방법
- 초기 설정: 감시 폴더 추가, 관리자 페이지 초기 비밀번호
- 문제 해결 표: 증상별 해결 방법 5가지
- 시스템 요구사항 표

**실제 배포 시 필요한 1회 설정:**
