### UC-APP01 — Electron 앱 설치 및 최초 실행

| 항목 | 내용 |
|------|------|
| **액터** | 일반 사용자 |
| **목적** | 인스톨러를 통해 앱을 설치하고, Java 없이 랜섬웨어 탐지 시스템을 실행한다 |
| **사전 조건** | `.dmg`(macOS) 또는 `.msi`(Windows) 인스톨러 파일이 다운로드되어 있다 |

**기본 흐름**

1. 사용자가 인스톨러를 실행한다
2. (Windows) SmartScreen 경고가 나타나면 "추가 정보 → 실행" 클릭
3. (macOS) "확인되지 않은 개발자" 경고가 나타나면 시스템 환경설정 → 보안에서 허용
4. 설치가 완료되고 앱이 자동 실행된다
5. Electron이 번들된 JRE로 Spring Boot JAR를 내부에서 시작한다
6. 로딩 화면이 표시되고, Spring Boot가 준비될 때까지 헬스체크(`/actuator/health`)를 500ms 간격으로 폴링한다
7. 60초 이내 헬스체크 성공 시 메인 화면(`localhost:8080`)이 로드된다
8. 앱이 시스템 트레이에 등록되고, OS 시작 시 자동 실행 설정이 저장된다

**예외 흐름**

- **E1**: 헬스체크가 60초 내에 성공하지 않으면 → 오류 화면(`error.html`) 표시, "재시도" 버튼 제공
- **E2**: AI 서버(`localhost:8001`)에 연결 불가 시 → 탐지 기능 비활성화, 나머지 기능은 정상 동작

**사후 조건**
- 앱이 정상 실행되고 대시보드가 표시된다
- 시스템 트레이에 WatchService 아이콘이 등록된다

### UC-APP02 — 앱 종료 및 트레이 최소화

| 항목 | 내용 |
|------|------|
| **액터** | 일반 사용자 |
| **목적** | 창을 닫아도 트레이에 상주하거나, 앱을 완전히 종료한다 |
| **사전 조건** | 앱이 실행 중이다 |

**기본 흐름 — 트레이 최소화**

1. 사용자가 창의 X(닫기) 버튼을 클릭한다
2. 창이 숨겨지고 시스템 트레이에 아이콘이 표시된다
3. Spring Boot 서버는 백그라운드에서 계속 실행된다
4. 사용자가 트레이 아이콘을 클릭하면 창이 다시 표시된다

**대안 흐름 — 앱 완전 종료**

1. 사용자가 트레이 아이콘 우클릭 → "종료" 클릭
2. Electron이 Spring Boot 프로세스에 SIGTERM을 전송한다
3. Spring Boot가 종료되면 Electron도 종료된다

**대안 흐름 — 시작 프로그램 등록/해제**

1. 사용자가 트레이 아이콘 우클릭 → "시작 프로그램 등록 해제" 클릭
2. `app.setLoginItemSettings({ openAtLogin: false })`가 호출된다
3. 다음 OS 재시작부터 앱이 자동 실행되지 않는다

### UC-W01 — 감시 시작

| 항목 | 내용 |
|------|------|
| **액터** | 일반 사용자 |
| **목적** | 등록된 폴더에 대해 실시간 파일 감시를 시작한다 |
| **사전 조건** | 감시 폴더가 1개 이상 등록되어 있다. 감시가 중지 상태다 |

**기본 흐름**

1. 사용자가 대시보드의 "감시 시작" 버튼을 클릭한다
2. `POST /api/watcher/start` 요청이 전송된다
3. 서버가 등록된 폴더들에 대해 NIO WatchService를 시작한다
4. 대시보드의 보호 상태 배지가 "SAFE"로 변경되고 버튼이 "감시 중지"로 바뀐다

**예외 흐름**

- **E1**: 등록된 폴더가 존재하지 않으면 → 토스트 오류 메시지 표시, 감시 미시작
- **E2**: 이미 감시 중이면 → 토스트로 안내

**사후 조건**
- 등록된 모든 폴더가 재귀적으로 감시된다
- 파일 이벤트(CREATE/MODIFY/DELETE) 발생 시 자동으로 `UC-D01`이 트리거된다

### UC-W03 — 즉시 검사(스캔) 실행

| 항목 | 내용 |
|------|------|
| **액터** | 일반 사용자 |
| **목적** | 현재 등록된 폴더의 파일을 즉시 전수 분석한다 |
| **사전 조건** | 감시 폴더가 1개 이상 등록되어 있다 |

**기본 흐름**

1. 사용자가 대시보드의 "즉시 검사" 버튼을 클릭한다
2. `POST /api/scan/start` 요청이 전송된다 (`{paths: [...], autoStartWatcher: true}`)
3. 서버가 `scanId`를 반환한다
4. 프론트엔드가 800ms 간격으로 `GET /api/scan/{scanId}/progress`를 폴링한다
5. 진행률 바가 0%에서 100%로 업데이트된다
6. 스캔 완료 시 `autoStartWatcher=true`이면 자동으로 감시가 시작된다
7. 대시보드 상태가 갱신된다

**대안 흐름 — 스캔 중단**

1. 사용자가 "일시 중지" 버튼을 클릭한다
2. `POST /api/scan/{scanId}/pause` 요청이 전송된다
3. 진행률 바가 멈추고 버튼이 "재개"로 바뀐다

**예외 흐름**

- **E1**: 스캔 중 오류 발생 시 → 진행률 바 오류 표시, 토스트 메시지

**사후 조건**
- 스캔된 파일들의 엔트로피·크기·확장자가 분석되어 스냅샷에 저장된다
- 스캔 결과가 로그에 기록된다

### UC-W04 — 감시 폴더 추가

| 항목 | 내용 |
|------|------|
| **액터** | 일반 사용자 |
| **목적** | 감시 대상 폴더를 등록한다 |
| **사전 조건** | 사용자가 대시보드 또는 설정 > 감시 폴더 페이지에 있다 |

**기본 흐름**

1. 사용자가 "폴더 추가" 버튼을 클릭한다
2. 백엔드 폴더 탐색기 모달이 열린다 (`GET /api/settings/folders/browse?path=/`)
3. 사용자가 폴더 목록에서 원하는 폴더로 이동하며 탐색한다 (클릭 시 경로 갱신)
4. 원하는 폴더를 선택하고 "이 폴더 선택" 버튼을 클릭한다
5. `POST /api/settings/folders` 요청 (`{name: 폴더명, path: 선택경로}`)이 전송된다
6. 폴더 목록이 갱신되고 토스트 성공 메시지가 표시된다

**대안 흐름 — 경로 직접 입력**

1. 사용자가 경로 입력 필드에 직접 경로를 타이핑한다
2. "추가" 버튼 클릭 시 서버로 전송된다

**예외 흐름**

- **E1**: 존재하지 않는 경로 → 서버에서 404, 토스트 오류
- **E2**: 이미 등록된 폴더 → 서버에서 409, 토스트 "이미 등록된 폴더"

**사후 조건**
- 새 폴더가 등록되고, 감시가 실행 중이라면 해당 폴더도 즉시 감시 대상에 추가된다

### UC-E01 — 예외 규칙 추가

| 항목 | 내용 |
|------|------|
| **액터** | 일반 사용자 |
| **목적** | 특정 경로 또는 확장자를 탐지 대상에서 제외한다 |
| **사전 조건** | 설정 > 예외 관리 페이지에 있다 |

**기본 흐름 — 경로 기반 (PATH)**

1. 사용자가 예외 종류를 "PATH"로 선택한다
2. "폴더 선택" 버튼을 클릭하여 폴더 탐색기 모달을 연다
3. 제외할 경로를 선택한다
4. 메모(선택)를 입력하고 "예외 추가" 버튼을 클릭한다
5. `POST /api/settings/exceptions` (`{type: "PATH", pattern: "/경로", memo: "..."}`}`) 전송
6. 목록에 새 규칙이 추가되고 토스트 성공 메시지가 표시된다

**대안 흐름 — 확장자 기반 (EXT)**

1. 사용자가 예외 종류를 "EXT"로 선택한다
2. 텍스트 필드에 확장자를 입력한다 (예: `.log`, `.tmp`)
3. 메모(선택)를 입력하고 "예외 추가" 버튼을 클릭한다
4. `POST /api/settings/exceptions` (`{type: "EXT", pattern: ".log", memo: "..."}`) 전송

**예외 흐름**

- **E1**: 동일한 패턴이 이미 존재하면 → 토스트 "이미 등록된 규칙"

**사후 조건**
- 해당 경로 또는 확장자의 파일 이벤트는 탐지 파이프라인에서 제외된다

### UC-D01 — 랜섬웨어 행위 자동 탐지 (시스템 자동)

| 항목 | 내용 |
|------|------|
| **액터** | 시스템 |
| **목적** | 파일 이벤트를 3초 단위로 집계하여 랜섬웨어 가능성을 판정하고 알림을 저장한다 |
| **사전 조건** | 감시가 시작된 상태다. AI 서버가 실행 중이다 |

**기본 흐름**

1. NIO WatchService가 감시 폴더에서 CREATE/MODIFY/DELETE 이벤트를 감지한다
2. 예외 규칙에 해당하는 이벤트는 즉시 필터링된다
3. `FileCollectorService`가 파일의 크기·엔트로피·확장자를 분석한다
4. 이벤트가 `EventWindowAggregator`의 3초 윈도우에 누적된다
5. 윈도우 종료 시 9개 피처가 계산된다:
   - `fileReadCount`, `fileWriteCount`, `fileDeleteCount`, `fileRenameCount`
   - `fileEncryptLikeCount`, `changedFilesCount`, `randomExtensionFlag`
   - `entropyDiffMean`, `fileSizeDiffMean`
6. `POST http://[AI서버]/api/analyze`로 피처 벡터가 전송된다
7. AI 서버가 XGBoost 모델로 점수를 계산한다:
   - 비-benign 클래스 확률 합산 → score
   - score ≥ 0.70 → DANGER / score ≥ 0.50 → WARNING / else → SAFE
8. `AiService`가 응답을 수신하고, score ≥ 0.50이면 랜섬웨어 패밀리 분류 API도 추가 호출한다
9. DANGER/WARNING 판정 시 Gemini API를 호출하여 대응 가이드를 생성한다 (선택, 타임아웃 10초)
10. `LogService.saveAsync()`로 이벤트 로그를 비동기 저장한다
11. `NotificationService.saveNotification()`으로 알림을 저장한다
12. 대시보드 상태가 갱신된다

**대안 흐름 — AI 서버 응답 없음**

- AI 서버 연결 타임아웃(5초) 또는 읽기 타임아웃(15초) 초과 시 → 해당 윈도우 판정을 UNKNOWN으로 기록하고 다음 윈도우 처리를 계속한다

**대안 흐름 — SCAN 이벤트**

- 즉시 검사(`UC-W03`)로 발생한 SCAN 이벤트 → 윈도우 집계에서 제외, 로그만 저장

**사후 조건**
- 이벤트 로그가 `event_log` 테이블에 저장된다
- 윈도우 단위 알림이 `notification` 테이블에 저장된다
- 대시보드의 최근 이벤트 목록이 갱신된다

| **사전 조건** | 관리자로 로그인된 상태다 |

**기본 흐름**

1. 관리자가 `/admin/alerts` 페이지에 접근한다
2. `GET /api/admin/alerts` 요청 → 전체 알림 목록 수신
3. 필터링 조건 적용 (날짜, 위험도, 키워드)

**대안 흐름 — 알림 삭제**

1. 삭제할 알림 옆 "삭제" 버튼 클릭
2. `DELETE /api/admin/alerts/{id}` 전송

### UC-A09 — 시스템 상태 조회

| 항목 | 내용 |
|------|------|
| **액터** | 관리자 |
| **목적** | 서버 운영 현황(DB 크기, 총 로그·알림 수, AI 서버 상태)을 확인한다 |
| **사전 조건** | 관리자로 로그인된 상태다 |

**기본 흐름**

1. 관리자가 `/admin/system` 페이지에 접근한다
2. `GET /api/admin/system` 요청이 전송된다
3. 다음 정보가 표시된다:
   - DB 크기 (bytes / MB)
   - 총 로그 수, 총 알림 수, 총 세션 수
   - 감시 실행 여부 (`watcherRunning`)
   - AI 서버 연결 상태 (`aiServerStatus`)

### UC-A10 — 사용 가이드 편집

| 항목 | 내용 |
|------|------|
| **액터** | 관리자 |
| **목적** | 사용자에게 표시되는 사용 가이드 내용을 수정한다 |
| **사전 조건** | 관리자로 로그인된 상태다 |

**기본 흐름**

1. 관리자가 `/admin/guide` 페이지에 접근한다
2. 현재 가이드 내용이 에디터에 로드된다 (`GET /api/guide`)
3. 내용을 수정하고 "저장" 버튼을 클릭한다
4. `PUT /api/admin/guide` (`{content}`) 요청이 전송된다
5. 토스트 성공 메시지가 표시된다

**사후 조건**
- 일반 사용자가 `/settings/guide` 페이지에서 수정된 가이드를 확인할 수 있다

## 유스케이스 간 관계

```
UC-W01 (감시 시작)
  └─ 트리거 → UC-D01 (자동 탐지)
                └─ 저장 → UC-N01 (알림 조회)
                       └─ 클릭 → UC-N02 (알림 상세)
                └─ 저장 → UC-L01 (로그 조회)
                       └─ 클릭 → UC-L02 (로그 상세)

UC-W03 (즉시 검사)
  └─ 완료 후 → UC-W01 (자동 감시 시작)

UC-E01 (예외 규칙 추가)
  └─ 이후 이벤트에서 UC-D01 탐지에서 제외

UC-F01 (피드백 제출)
  └─ 관리자가 → UC-A04 (피드백 조회·삭제)

UC-A01 (관리자 로그인)
  └─ 선행 조건 → UC-A03~A10 모두
```
