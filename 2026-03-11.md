## 1. 피처 계산 정확도 개선 (크로스 플랫폼)

NIO WatchService 이벤트(CREATE/MODIFY/DELETE)만으로 계산 가능한 범위 내에서 개선.

| 파일 | 변경 내용 |
|------|----------|
| `application.yml` | `random-ext.whitelist` 확장: html, htm, json, yaml, yml, java, pptx, ppt, csv, flac, webm, webp, tiff, bmp, conf, toml, sqlite, mpeg, opus 추가 (기존 14개 → 33개). 개발자·미디어 파일 오탐 방지 |
| `analytics/EventWindowAggregator.java` | `fileWriteCount`: 대용량 CREATE 이벤트(sizeAfter ≥ encryptMinSizeBytes)도 write로 카운트 — 암호화된 복사본 생성 패턴 탐지 |
| `analytics/EventWindowAggregator.java` | `fileEncryptLikeCount`: 스트림 암호 in-place 패턴 탐지 추가 — MODIFY(엔트로피↑, 크기·확장자 동일) → 같은 윈도우 내 DELETE 시퀀스 감지 |

### 개선 이유

- **whitelist 부족**: 기존 14개 확장자만 안전 목록에 있어서 .json, .html, .java 등 흔한 파일이 suspicious로 오탐
- **CREATE write 미포함**: 랜섬웨어가 암호화 복사본을 CREATE할 때 writeCount가 증가하지 않아 피처값 과소 추정
- **스트림 암호 패턴 미탐**: RC4 등 스트림 암호로 in-place 암호화 시 크기·확장자 변화 없어서 기존 encrypt-like 판정 통과 불가. DELETE 시퀀스로 보완

### 빌드 결과

```
./gradlew build -x test --no-daemon → BUILD SUCCESSFUL
```

## 2. fileReadCount 개선

**원리**: "파일을 삭제하려면 반드시 먼저 접근해야 한다" → DELETE 이벤트를 touch session에 포함.

| 파일 | 변경 내용 |
|------|----------|
| `analytics/EventWindowAggregator.java` | `fileReadCount`: DELETE 이벤트 발생 시 해당 파일 경로를 touch session에 포함. 5분 세션 중복 방지 로직 동일 적용 |

### 개선 이유

- **READ 이벤트 부재**: NIO WatchService는 READ 이벤트를 제공하지 않아 fileReadCount가 MODIFY 기반으로만 추정됨
- **랜섬웨어 패턴 미반영**: `READ → CREATE(암호화본) → DELETE(원본)` 패턴에서 DELETE가 touchCount에 포함되지 않아 접근 수 과소 추정
- **DELETE = 접근 증거**: 파일 삭제는 반드시 접근 이후에 발생하므로 DELETE를 touch 증거로 사용하는 것이 합리적

### 개선 효과

| 상황 | 이전 fileReadCount | 이후 fileReadCount |
|------|-------------------|-------------------|
| 파일 100개 MODIFY | 100 | 100 (변화 없음) |
| 파일 50개 READ → DELETE (랜섬웨어) | 0 | 50 (개선) |
| rename 10개 (DELETE+CREATE 쌍) | 0 | 10 (개선) |
| MODIFY 30개 + DELETE 20개 | 30 | 최대 50 (개선) |

### 빌드 결과

```
./gradlew build -x test --no-daemon → BUILD SUCCESSFUL
```
