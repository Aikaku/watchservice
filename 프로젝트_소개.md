# WatchService Agent
## 랜섬웨어 실시간 탐지 시스템

---

## Motivation | 동기

랜섬웨어는 사용자의 파일을 암호화한 뒤 복호화 대가로 금전을 요구하는 악성코드로,
피해가 발생한 후에는 복구가 사실상 불가능한 경우가 많다.
기존 백신 소프트웨어는 알려진 시그니처 기반으로 동작하기 때문에
**신종·변종 랜섬웨어**에 대한 대응이 느리다는 한계가 있다.

이 프로젝트는 다음 문제의식에서 출발했다.

> "파일이 암호화되는 **행위 자체**를 실시간으로 감지할 수 있다면,
> 시그니처 없이도 신종 랜섬웨어를 탐지할 수 있지 않을까?"

파일의 **엔트로피 변화**, **확장자 변경**, **대량 삭제·rename 패턴** 등
랜섬웨어가 암호화 과정에서 공통적으로 남기는 흔적을 머신러닝으로 학습해,
시그니처에 의존하지 않는 행위 기반 탐지 시스템을 구축하고자 했다.

---

## Algorithm | 알고리즘

### 전체 탐지 파이프라인

```
[파일 이벤트 발생]
        │  Java NIO WatchService (CREATE / MODIFY / DELETE)
        ▼
┌─────────────────────────────────┐
│  FileCollectorService           │
│  - Shannon 엔트로피 계산        │
│    (3구간 샘플링: 앞40/중40/끝20%)│
│  - 파일 크기 측정               │
│  - 확장자 추출                  │
│  - 스냅샷 비교 (before / after) │
└────────────────┬────────────────┘
                 │
                 ▼
┌─────────────────────────────────┐
│  EventWindowAggregator (3초)    │
│  9개 피처 벡터 계산             │
└────────────────┬────────────────┘
                 │  AiPayload (9 features)
                 ▼
┌─────────────────────────────────┐
│  XGBoost AI 서버 (Python)       │
│  POST /api/analyze              │
│  → score, label, topFamily      │
└────────────────┬────────────────┘
                 │
                 ▼
        SAFE / WARNING / DANGER
                 │
        DANGER → Gemini LLM 대응 가이드 생성
                 │
                 ▼
     알림 저장 + 이메일 발송 + 전체화면 경보
```

---

### XGBoost 입력 피처 (9개)

| # | 피처명 | 설명 |
|---|--------|------|
| 1 | `fileReadCount` | 세션 기반 파일 접근 수 |
| 2 | `fileWriteCount` | 내용 변경(크기·엔트로피 변화) 기반 쓰기 수 |
| 3 | `fileDeleteCount` | DELETE 이벤트 수 |
| 4 | `fileRenameCount` | DELETE+CREATE 쌍 매칭으로 탐지된 rename 수 |
| 5 | `fileEncryptLikeCount` | 엔트로피 급증 + 크기·확장자 변화 파일 수 |
| 6 | `changedFilesCount` | 변경된 고유 파일 수 |
| 7 | `randomExtensionFlag` | 의심 확장자 파일 2개 이상이면 1 |
| 8 | `entropyDiffMean` | 유의미한 엔트로피 변화 평균 |
| 9 | `fileSizeDiffMean` | 파일 크기 변화 평균 (bytes) |

---

### Shannon 엔트로피 계산

파일이 암호화되면 데이터의 무작위성(엔트로피)이 급격히 증가한다.

```
Shannon Entropy H = -Σ p(x) × log₂(p(x))

- 일반 텍스트 파일:  H ≈ 3~5 bit
- 암호화된 파일:     H ≈ 7.5~8.0 bit (이론 최대값 8)
```

대용량 파일의 성능을 위해 **3구간 샘플링** 적용:
- 앞 40% + 중간 40% + 끝 20% 구간을 각각 4KB 샘플링

---

### Rename 탐지 (DELETE + CREATE 쌍 매칭)

Java NIO WatchService는 파일 rename을 DELETE + CREATE 두 이벤트로 보고한다.
두 이벤트를 **점수 기반 매칭**으로 rename으로 판정:

| 조건 | 점수 |
|------|------|
| 파일 stem 동일 (확장자만 변경) | +2 |
| 파일 크기 동일 | +2 |
| 경로가 `원본.ext.wncry` 형태 (ext-append) | +3 |
| 확장자 동일 | +1 |
| 점수 ≥ 3, 시간 간격 ≤ 2초 | rename으로 판정 |

---

### 탐지 임계값

| 점수 (score) | 레벨 |
|-------------|------|
| 0.70 이상 | DANGER |
| 0.50 ~ 0.70 | WARNING |
| 0.50 미만 | SAFE |

## Development | 요약

### 주요 기능

| 기능 | 설명 |
|------|------|
| 실시간 파일 감시 | Java NIO WatchService로 감시 폴더 재귀 감시 |
| 행위 기반 AI 탐지 | XGBoost 9개 피처 벡터로 랜섬웨어 판정 |
| 랜섬웨어 패밀리 분류 | WannaCry, LockBit, REvil 등 13종 분류 |
| 대응 가이드 생성 | DANGER 탐지 시 Gemini LLM이 맞춤 대응 방법 제시 |
| 긴급 전체화면 경보 | Electron이 DANGER 폴링 → 전체화면 경보창 팝업 |
| 이메일 알림 | DANGER 탐지 시 설정된 이메일로 경보 메일 자동 발송 |
| 수동 스캔 | 감시 폴더 전체를 즉시 검사, 진행률 실시간 표시 |
| 파일 권한 감사 | others 쓰기·실행 권한 파일 탐지 (macOS/Linux) |
| 감시 스케줄 | 특정 요일·시간대에만 감시 활성화 |
| 예외 규칙 | 경로·확장자 화이트리스트 등록으로 오탐 방지 |
| 관리자 페이지 | 전체 세션 로그·알림 조회, 공지사항·가이드 관리 |

### 시스템 구성

```
사용자 PC
  └── Electron 앱
        ├── Spring Boot 백엔드 (Java, :8080)
        │     ├── React 프론트엔드 (정적 파일 내장)
        │     └── SQLite DB (log.db)
        └── JRE 번들 내장 (Java 설치 불필요)

클라우드 (Railway)
  └── Python AI 서버 (FastAPI + XGBoost, :8001)

외부 API
  ├── Google Gemini API (대응 가이드 생성)
  └── GitHub Releases API (업데이트 확인)
```

### 배포 형태

- **macOS**: `.dmg` 인스톨러 (Intel / Apple Silicon)
- **Windows**: `.msi` 인스톨러
- AI 서버는 Railway 클라우드에 상시 배포, 별도 설치 불필요
