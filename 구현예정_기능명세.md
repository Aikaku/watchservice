## 1. 이메일 알림

### 목적
DANGER 탐지 시 설정한 이메일 주소로 경고 메일을 자동 발송한다. 모니터를 보지 않는 상황에서도 즉각 인지할 수 있다.

### 동작 흐름
```
NotificationService.saveNotification() 호출
  → ai_label == DANGER 이면
  → EmailNotificationService.sendAlert(notification) 비동기 호출
  → SMTP 서버로 MimeMessage 전송
```

### 백엔드 구현
- `spring-boot-starter-mail` 의존성 추가
- `application.yml`에 SMTP 설정 추가
  ```yaml
  spring.mail:
    host: ${SMTP_HOST:smtp.gmail.com}
    port: ${SMTP_PORT:587}
    username: ${SMTP_USERNAME:}
    password: ${SMTP_PASSWORD:}
    properties.mail.smtp.starttls.enable: true
  ```
- `EmailNotificationService` — `JavaMailSender`로 HTML 이메일 생성 + `@Async` 비동기 발송
- 이메일 내용: 탐지 레벨(DANGER), 탐지 시각, 감시 폴더, 변경 파일 수, Gemini 대응 가이드 요약

### 프론트엔드 구현
- 설정 페이지에 수신 이메일 주소 입력 필드 + 테스트 발송 버튼
- `SettingApi.js`에 이메일 설정 저장/조회 API 추가

### DB 변경
- `settings` 테이블에 `alert_email` 컬럼 추가

## 2. 수동 스캔

### 목적
감시 폴더를 즉시 전수 스캔하여 현재 상태를 분석한다. 실시간 감시와 별개로 원하는 시점에 직접 점검할 수 있다.

### 동작 흐름
```
사용자가 "스캔 시작" 버튼 클릭
  → POST /api/scan/start
  → ScanService.startScan() 비동기 실행
  → Files.walk()로 감시 폴더 순회
  → 각 파일마다 FileCollectorService.analyze() 호출 (엔트로피, 크기, 확장자 계산)
  → GET /api/scan/status 폴링으로 진행률 확인
  → 스캔 완료 시 결과 리포트 반환
```

### 백엔드 구현
- `ScanService` — `FileCollectorService` 재활용. `ScanApi.js`가 이미 일부 구현됨
- `POST /api/scan/start` — 비동기 스캔 시작, `scanId` 반환
- `GET /api/scan/status/{scanId}` — 진행률(%), 스캔된 파일 수, 의심 파일 수 반환
- `GET /api/scan/result/{scanId}` — 완료 후 결과 조회 (의심 파일 목록, 엔트로피 분포)
- 스캔 결과는 `SCAN` 이벤트 타입으로 `event_log`에 저장 (기존 SCAN 타입 재활용)

### 프론트엔드 구현
- 대시보드 또는 설정 페이지에 "지금 스캔" 버튼
- 진행 중 프로그레스 바 표시
- 완료 후 결과 모달 (의심 파일 목록 테이블)

## 3. 오탐 신고

### 목적
정상 파일이 랜섬웨어로 오탐됐을 때 신고하면 해당 파일 경로가 예외 규칙에 자동 등록된다. 이후 동일 파일에서 알림이 반복 발생하지 않는다.

### 동작 흐름
```
알림 상세 페이지에서 "오탐 신고" 버튼 클릭
  → PATCH /api/notifications/:id/false-positive
  → notification.is_false_positive = true 업데이트
  → SettingsService.addException(filePath) 자동 호출
  → 예외 규칙 DB에 해당 파일 경로 등록
```

### 백엔드 구현
- `PATCH /api/notifications/:id/false-positive` 엔드포인트 추가
- `notification` 테이블에 `is_false_positive INTEGER DEFAULT 0` 컬럼 추가
- `SettingsService.addException()` 기존 메서드 재활용

### 프론트엔드 구현
- 알림 상세 페이지에 "오탐 신고" 버튼 추가
- 신고된 알림은 목록에서 취소선 또는 회색 처리로 시각적 구분
- 신고 후 "예외 규칙에 등록됐습니다" 토스트 표시

## 4. 사용자 정의 탐지 규칙

### 목적
AI 판정과 별개로 사용자가 직접 탐지 조건을 정의한다. "3초 안에 파일 10개 이상 수정되면 WARNING" 같은 커스텀 규칙을 UI에서 설정할 수 있다.

### 동작 흐름
```
EventWindowAggregator.computeWindowStats() 실행 (9개 피처 계산)
  → CustomRuleService.evaluate(windowStats) 호출
  → 활성화된 커스텀 규칙 전체 체크
  → 커스텀 레벨 결정 (규칙 미충족 시 null)
  → AiService.analyze() 호출
  → finalLabel = max(customLabel, aiLabel)  // DANGER > WARNING > SAFE
```

### 조건에 사용 가능한 메트릭 (9개 피처)
| 메트릭 | 설명 |
|--------|------|
| `fileReadCount` | 파일 읽기 수 |
| `fileWriteCount` | 파일 쓰기 수 |
| `fileDeleteCount` | 파일 삭제 수 |
| `fileRenameCount` | 파일 rename 수 |
| `fileEncryptLikeCount` | 암호화 의심 파일 수 |
| `changedFilesCount` | 변경된 고유 파일 수 |
| `randomExtensionFlag` | 의심 확장자 감지 (0/1) |
| `entropyDiffMean` | 엔트로피 변화 평균 |
| `fileSizeDiffMean` | 파일 크기 변화 평균 |

### DB 구조
```sql
CREATE TABLE custom_rule (
    id             INTEGER PRIMARY KEY AUTOINCREMENT,
    name           TEXT NOT NULL,
    condition_json TEXT NOT NULL,
    action         TEXT DEFAULT 'WARNING',  -- WARNING / DANGER
    enabled        INTEGER DEFAULT 1,
    created_at     TEXT NOT NULL
);
```

`condition_json` 예시:
```json
{ "metric": "fileDeleteCount", "operator": "gte", "threshold": 5 }
```

### 백엔드 구현
- `CustomRuleRepository`, `CustomRuleService` 신규 생성
- `GET/POST/PUT/DELETE /api/settings/custom-rules` CRUD API
- `EventWindowAggregator`에서 AI 호출 전 `CustomRuleService.evaluate()` 삽입

### 프론트엔드 구현
- 설정 페이지 커스텀 규칙 탭
- 규칙 목록 테이블 (이름, 조건 요약, 발동 레벨, 활성화 토글)
- 규칙 추가/수정 모달 (메트릭 드롭다운 → 연산자 선택 → 임계값 입력)

### 주의사항
- 윈도우 크기는 현재 고정값 기준이므로 규칙별 윈도우 설정은 지원하지 않는다
- 임계값을 너무 낮게 설정하면 오탐이 폭발할 수 있으므로 입력 시 경고 메시지 표시 권장

## 5. Electron 긴급 전체 화면 경보

### 목적
DANGER 탐지 시 Electron 앱이 전체 화면 긴급 알림 창을 강제 표시한다. 백그라운드 감시 중이거나 다른 작업을 하고 있어도 즉각 인지할 수 있다.

### 동작 흐름
```
main.js 폴링 (2초 간격)
  → GET /api/notifications?label=DANGER&limit=1&after={lastCheckedId}
  → 새 DANGER 알림 감지 시
  → new BrowserWindow({ fullscreen: true, alwaysOnTop: true }) 생성
  → emergency.html 로드 + 경보음 재생
  → 사용자가 "확인" 버튼 클릭 시 창 닫힘
```

### Electron 구현 (`main.js`)
```javascript
let emergencyWindowOpen = false;

function checkForDanger() {
  fetch(`http://localhost:8080/api/notifications?label=DANGER&after=${lastId}`)
    .then(res => res.json())
    .then(data => {
      if (data.length > 0 && !emergencyWindowOpen) {
        showEmergencyWindow(data[0]);
      }
    });
}

setInterval(checkForDanger, 2000);
```

- `emergency.html` — 빨간 전체화면 배경 + 탐지 정보 + "즉시 대응하세요" 메시지 + 확인 버튼
- `shell.beep()` 또는 `<audio>` 태그로 경보음 재생
- 이미 창이 열려 있으면 중복 생성 방지 (`emergencyWindowOpen` 플래그)

## 6. 감시 폴더 삭제·이름 변경 감지

### 목적
감시 중인 루트 폴더 자체가 삭제되거나 이름이 변경될 때 DANGER 경보를 발생시킨다. 랜섬웨어가 폴더 구조 자체를 파괴하는 패턴을 탐지한다.

### 동작 흐름
```
WatcherService 초기화 시
  → 감시 폴더 자체를 감시하는 것에 더해
  → 감시 폴더의 상위 디렉토리도 WatchService에 등록
  → 상위 디렉토리에서 감시 폴더명과 일치하는 DELETE 이벤트 발생 시
  → DANGER 레벨 알림 즉시 생성
  → 감시 재등록 시도 (폴더 재생성 감지 후 자동 재등록)
```

### 백엔드 구현
- `WatcherService`에서 감시 폴더 등록 시 상위 디렉토리도 함께 등록
- 이벤트 처리 시 `deletedPath == watchedRootFolder` 조건 체크
- 일치하면 `NotificationService.saveNotification()` 직접 호출 (AI 판정 우회, 즉시 DANGER)
- 폴더 재생성 이벤트(CREATE) 감지 시 `WatcherService`에 자동 재등록

## 7. 탐지 리포트 PDF 생성

### 목적
기간별 탐지 현황(요약 통계, 트렌드 차트, 상세 목록)을 PDF 파일로 내보낸다. CERT/SOC 보고 또는 정기 보안 리뷰에 활용한다.

### 동작 흐름
```
GET /api/report?from=2026-01-01&to=2026-03-31&format=pdf
  → ReportService.generate(from, to)
  → NotificationRepository + LogRepository에서 기간 데이터 조회
  → PDFBox로 문서 생성 (요약 테이블 + 차트 이미지 + 상세 목록)
  → PDF 바이트 스트림 반환
```

### 백엔드 구현
- `Apache PDFBox` 의존성 추가 (`org.apache.pdfbox:pdfbox:3.x`)
- `ReportService` 신규 생성
  - 1페이지: 기간 요약 (SAFE/WARNING/DANGER 건수, 가장 많이 탐지된 패밀리)
  - 2페이지: 일별 탐지 건수 차트 (PDFBox로 직접 그리기 또는 Base64 이미지 삽입)
  - 3페이지~: 알림 상세 목록 (시각, 레벨, 패밀리, 파일 경로)
- `GET /api/report` — `Content-Disposition: attachment; filename=report.pdf` 헤더 포함

### 프론트엔드 구현
- 로그 또는 알림 페이지에 "리포트 내보내기" 버튼
- 기간 선택 datepicker
- 클릭 시 파일 다운로드 트리거

## 8. 알림 심각도별 소리 알림

### 목적
탐지 발생 시 심각도에 따라 다른 소리를 재생하여 화면을 보지 않아도 즉각 인지할 수 있게 한다.

### 동작 흐름
```
프론트엔드에서 새 알림 감지 (폴링 또는 SSE)
  → ai_label 확인
  → DANGER: 강한 경고음 재생
  → WARNING: 부드러운 알림음 재생
  → SAFE: 소리 없음
```

### 프론트엔드 구현
- `AudioAlert.jsx` 컴포넌트 — Web Audio API (`AudioContext`) 또는 `<audio>` 태그
- 소리 파일 3종: `alert_danger.mp3`, `alert_warning.mp3` (`public/sounds/`)
- 소리 ON/OFF 설정 + 볼륨 조절 → `localStorage` 저장
- `MainLayout`에 `AudioAlert` 마운트 (전 페이지 공통 적용)

### 주의사항
- 브라우저 자동재생 정책(Autoplay Policy)으로 인해 사용자 인터랙션 없이는 소리 재생이 차단될 수 있다. 앱 로드 후 최초 클릭 이벤트 발생 시 `AudioContext`를 resume 상태로 초기화해야 한다.

## 9. 감시 스케줄 설정

### 목적
특정 시간대에만 감시를 활성화한다. 야간 백업, 업무 시간 외 점검 등 특정 시간대에 집중 감시하거나 불필요한 알림을 줄일 수 있다.

### 동작 흐름
```
@Scheduled(fixedDelay = 60000)  // 1분마다
ScheduleCheckService.check()
  → DB에서 스케줄 설정 조회
  → 현재 시각이 활성 범위 내이면 WatcherService.resume()
  → 활성 범위 밖이면 WatcherService.pause()
```

### 백엔드 구현
- `WatcherService`에 `pause()` / `resume()` 메서드 추가
  - `pause()`: WatchKey 등록 해제 + 감시 스레드 대기
  - `resume()`: 감시 폴더 재등록 + 감시 재개
- `schedule` DB 테이블
  ```sql
  CREATE TABLE watch_schedule (
      id         INTEGER PRIMARY KEY AUTOINCREMENT,
      start_time TEXT NOT NULL,   -- "09:00"
      end_time   TEXT NOT NULL,   -- "18:00"
      days_of_week TEXT NOT NULL, -- "1,2,3,4,5" (월~금)
      enabled    INTEGER DEFAULT 1
  );
  ```
- `GET/PUT /api/settings/schedule` API

### 프론트엔드 구현
- 설정 페이지에 스케줄 섹션
- 시작/종료 시간 입력 + 요일 체크박스
- 스케줄 활성화 토글
- 현재 감시 상태(활성/일시정지) 표시

### 지원 패밀리 (현재 `AiService`에 하드코딩된 목록 기준)
LockBit, WannaCry, REvil, Ryuk, Conti, BlackCat, Clop, Hive, BlackMatter, DarkSide 등

- `src/main/resources/families/` 폴더에 패밀리별 JSON 파일 내장
  ```json
  {
    "name": "WannaCry",
    "firstSeen": "2017",
    "attackMethod": "SMB 취약점(EternalBlue) 익스플로잇으로 네트워크 전파",
    "targets": ["Windows XP~7", "의료기관", "물류"],
    "knownExtensions": [".WCRY", ".WNCRY", ".WNCRYT"],
    "mitigation": "MS17-010 패치 적용, SMB 포트 445 차단"

  ```
- `FamilyInfoController` — `GET /api/families/{name}`

### 프론트엔드 구현
- 알림 상세 페이지에서 패밀리명을 클릭 가능한 링크로 표시
- 클릭 시 상세 정보 모달 (공격 방식, 주요 타깃, 알려진 확장자, 대응 방법)
- 정보 없는 패밀리는 "정보 없음" 처리

## 11. 파일 확장자 분포 차트

### 목적
탐지된 의심 이벤트에서 가장 많이 등장한 파일 확장자 상위 10개를 차트로 시각화한다. 어떤 파일 유형이 집중 공격받는지 파악할 수 있다.

### 동작 흐름
```
GET /api/logs/extension-stats?from=&to=
  → event_log에서 파일 경로의 확장자 추출
  → 확장자별 COUNT GROUP BY ORDER BY COUNT DESC LIMIT 10
  → 프론트엔드 recharts PieChart / BarChart로 렌더링
```

### 백엔드 구현
- `LogRepository`에 확장자 집계 쿼리 추가
  ```sql
  SELECT LOWER(SUBSTR(file_path, INSTR(file_path, '.') + 1)) AS ext,
         COUNT(*) AS cnt
  FROM event_log
  WHERE created_at BETWEEN :from AND :to
  GROUP BY ext
  ORDER BY cnt DESC
  LIMIT 10
  ```
- `GET /api/logs/extension-stats` 엔드포인트 추가

### 프론트엔드 구현
- 대시보드 또는 로그 페이지에 차트 컴포넌트 추가
- `recharts` PieChart (확장자별 비율) 또는 BarChart (건수)
- 기간 필터 연동

## 12. 자주 변경되는 파일 TOP 10

### 목적
감시 기간 동안 이벤트가 가장 많이 발생한 파일 경로 TOP 10을 표시한다. 랜섬웨어가 집중적으로 공격하는 파일을 빠르게 파악할 수 있다.

### 동작 흐름
```
GET /api/logs/top-files?limit=10&from=&to=
  → event_log에서 file_path별 COUNT GROUP BY ORDER BY COUNT DESC
  → 프론트엔드 순위 테이블로 렌더링
```

### 백엔드 구현
- `LogRepository`에 집계 쿼리 추가
  ```sql
  SELECT file_path, COUNT(*) AS cnt
  FROM event_log
  WHERE created_at BETWEEN :from AND :to
  GROUP BY file_path
  ORDER BY cnt DESC
  LIMIT :limit
  ```
- `GET /api/logs/top-files` 엔드포인트 추가

### 프론트엔드 구현
- 대시보드 또는 로그 페이지에 TOP 10 테이블 (순위, 파일 경로, 이벤트 수)
- 파일 경로 클릭 시 해당 파일의 로그 필터링 연동

## 13. 파일 권한 보안 감사

### 목적
감시 폴더 내 파일 중 다른 사용자(others)에게 쓰기·실행 권한이 부여된 파일을 조회하고, 권한 수정 기능을 제공한다. 랜섬웨어 탐지보다는 파일 보안 상태 감사 목적이다.

### 동작 흐름
```
수동 트리거 또는 @Scheduled(1시간마다)
  → Files.walk()로 감시 폴더 내 파일 순회
  → OS 분기
    → macOS/Linux: Files.getPosixFilePermissions() → OTHERS_WRITE / OTHERS_EXECUTE 체크
    → Windows: JNA(jna-platform) → SID S-1-1-0(Everyone) ACL 항목에서 WRITE_DATA / EXECUTE 체크
  → 해당 파일 목록 DB 저장
  → GET /api/security/permission-audit 로 조회
```

### 백엔드 구현
- `build.gradle`에 JNA 의존성 추가 (Windows 지원용)
  ```groovy
  implementation 'net.java.dev.jna:jna:5.14.0'
  implementation 'net.java.dev.jna:jna-platform:5.14.0'
  ```
- `PermissionAuditService` — OS 분기 처리
  ```java
  boolean isPosix = FileSystems.getDefault()
      .supportedFileAttributeViews().contains("posix");
  ```
- macOS/Linux: `Files.getPosixFilePermissions()` + `OTHERS_WRITE`, `OTHERS_EXECUTE` 체크
- Windows: `AclFileAttributeView` + SID `S-1-1-0` 직접 비교 (언어 독립적)
- 권한 수정: `Files.setPosixFilePermissions()`으로 others 권한 제거
- `GET /api/security/permission-audit` — 위험 파일 목록 반환
- `POST /api/security/permission-audit/fix/{fileId}` — others 권한 제거

### 프론트엔드 구현
- 설정 또는 보안 페이지에 "권한 감사 실행" 버튼
- 결과 테이블: 파일 경로, 현재 권한(755 형식), 위험 항목 강조, 권한 수정 버튼
- Windows에서는 POSIX 표기 대신 ACL 설명으로 표시

### 주의사항
- 권한 수정은 소유자 또는 관리자만 가능. 권한 부족 시 오류 처리 필요
- 격리 폴더 경로는 감사 대상에서 제외 권장
