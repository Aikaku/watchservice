## 1. 테스트 환경 준비

### 1.1 필수 환경

| 항목 | 요구사항 |
|------|---------|
| Java | 17 이상 |
| Node.js | 18 이상 |
| Python | 3.9 이상 (AI 서버 테스트 시) |
| 운영체제 | macOS / Windows |

### 1.2 환경변수 설정

프로젝트 루트(`watchservice_be/`) 또는 시스템 환경변수로 설정:

```bash
# 관리자 인증 (BCrypt 해시 필수)
# 해시 생성 방법 (Java): new BCryptPasswordEncoder().encode("admin1234")
# 또는 아래 파이썬 한 줄:
# python3 -c "import bcrypt; print(bcrypt.hashpw(b'admin1234', bcrypt.gensalt()).decode())"
ADMIN_USERNAME=admin
ADMIN_PASSWORD=$2a$10$xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx

# 일반 사용자 인증 (테스트 시 활성화)
USER_AUTH_ENABLED=true
USER_PASSWORD=user1234

# Gemini API (선택)
GEMINI_API_KEY=your_gemini_api_key

# 이메일 알림 (Gmail 예시)
SMTP_HOST=smtp.gmail.com
SMTP_PORT=587
SMTP_USERNAME=your_account@gmail.com
SMTP_PASSWORD=your_app_password   # Gmail 앱 비밀번호 (2단계 인증 필요)

# AI 서버 (로컬 실행 시 기본값 사용 가능)
AI_ANALYZE_URL=http://localhost:8001/api/analyze
AI_FAMILY_URL=http://localhost:8001/predict
```

### 1.3 서비스 실행 순서

```
1. AI 서버 (선택)
   cd ai_server_deploy
   python api_server.py

2. 백엔드
   cd watchservice_be
   ./gradlew bootRun

3. 프론트엔드 (개발 서버)
   cd watchservice_fe
   npm start

4. Electron (선택)
   cd watchservice_electron
   npm start
```

> **참고**: 프론트엔드 단독 테스트 시 Electron 불필요. 이메일 테스트 시 AI 서버 불필요.

## 2. 기본 기동 확인

**목적**: 서버 정상 기동, DB 파일 생성, 프론트 렌더링 확인

### 단계

1. `./gradlew bootRun` 실행
2. 로그에서 아래 메시지 확인:
   ```
   [SettingsRepository] watched_folder 테이블 초기화 완료
   [SettingsRepository] exception_rule 테이블 초기화 완료
   [SettingsRepository] app_settings 테이블 초기화 완료
   [NotificationRepository] notification 테이블 초기화 완료
   ```
3. 프로젝트 루트에 `log.db` 파일이 생성됐는지 확인
4. 브라우저에서 `http://localhost:3000` 접속
5. 메인 대시보드(`/`) 렌더링 확인 (로딩 에러 없음)

### 기대 결과

- 서버 콘솔에 ERROR 없이 `Started WatchserviceAgentApplication` 출력
- `log.db` 파일 존재
- 대시보드 정상 표시

## 3. 관리자 인증 (BCrypt 강제)

**목적**: 평문 비밀번호로 관리자 로그인이 차단되는지 확인

### 3.1 기본값(평문) 차단 테스트

1. `ADMIN_PASSWORD` 환경변수를 **설정하지 않은 채** 서버 기동
2. 서버 로그에서 아래 메시지 확인:
   ```
   [SECURITY] 관리자 비밀번호가 기본값(123456789)입니다.
   [SECURITY] 환경변수 ADMIN_PASSWORD를 BCrypt 해시로 설정하세요.
   ```
3. `http://localhost:3000/admin/login` 접속
4. ID: `admin`, PW: `123456789` 입력 후 로그인 시도

**기대 결과**: 로그인 실패 (401 반환), 백엔드 로그에 `admin.password가 BCrypt 해시가 아닙니다. 로그인이 차단됩니다.` 출력

### 3.2 BCrypt 해시 로그인 성공 테스트

1. BCrypt 해시 생성:
   ```bash
   # Python
   python3 -c "import bcrypt; print(bcrypt.hashpw(b'admin1234', bcrypt.gensalt()).decode())"
   # 출력 예: $2b$12$xxxx...
   ```
2. `ADMIN_PASSWORD=$2b$12$xxxx...` 환경변수 설정 후 서버 재기동
3. `http://localhost:3000/admin/login` 에서 `admin` / `admin1234` 로그인

**기대 결과**: 로그인 성공 → `/admin/main` 리다이렉트

### 3.3 세션 타임아웃 테스트 (선택)

1. 관리자 로그인 후 `/admin/main` 접속 확인
2. `application.yml`의 `server.servlet.session.timeout`을 `1m`으로 임시 변경
3. 1분 대기 후 관리자 API 재호출
4. 401 반환 확인 → 프론트가 `/admin/login`으로 리다이렉트하는지 확인

## 4. 일반 사용자 인증

**목적**: `USER_AUTH_ENABLED=true` 시 일반 사용자 페이지 보호 확인

### 단계

1. 환경변수 설정:
   ```
   USER_AUTH_ENABLED=true
   USER_PASSWORD=user1234
   ```
2. 서버 재기동
3. `http://localhost:3000/notifications` 직접 접속
4. `/login`으로 리다이렉트 되는지 확인
5. `/login` 페이지에서 `user1234` 입력 후 로그인
6. `/notifications` 정상 접근 확인
7. `POST http://localhost:8080/api/user/logout` 호출 (또는 브라우저 세션 삭제)
8. `/notifications` 재접속 → 다시 `/login` 리다이렉트 확인

**목적**: 미인증 상태에서 URL 직접 입력 시 리다이렉트 확인

### 5.1 관리자 페이지 차단

1. 관리자 로그아웃 상태에서 `http://localhost:3000/admin/main` 직접 입력

### 5.2 일반 사용자 페이지 차단 (`USER_AUTH_ENABLED=true`)

1. 미로그인 상태에서 `http://localhost:3000/logs` 직접 입력
2. `/login`으로 즉시 리다이렉트 확인

## 6. GlobalExceptionHandler 응답 형식

**목적**: API 오류 시 JSON 형식 응답 반환 확인

### 단계

1. 존재하지 않는 알림 ID 조회:
   ```bash
   curl http://localhost:8080/notifications/99999
   ```
2. 응답 형식 확인:
   ```json
   { "success": false, "code": "NOT_FOUND", "message": "notification not found: id=99999" }
   ```
3. 잘못된 데이터로 감시 폴더 추가:
   ```bash
   curl -X POST http://localhost:8080/settings/folders \
     -H "Content-Type: application/json" \
     -d '{"path": ""}'
   ```
4. 응답:
   ```json
   { "success": false, "code": "BAD_REQUEST", "message": "path: 공백일 수 없습니다" }
   ```

**기대 결과**: HTML 페이지나 Spring 기본 에러 형식이 아닌 `{ success, code, message }` JSON 반환

## 8. 감시 폴더 관리

**목적**: 폴더 추가/삭제, 브라우저 탐색 모달 정상 동작 확인

### 단계

1. `http://localhost:3000/settings/folders` 접속
2. "폴더 선택" 버튼 클릭 → 서버 파일시스템 탐색 모달 팝업 확인
3. 모달에서 원하는 폴더 탐색 → "이 폴더 선택" 클릭
4. 감시 폴더 목록에 추가됐는지 확인
5. 추가된 폴더 옆 "삭제" 버튼 클릭 → 목록에서 제거 확인
6. 페이지 새로고침 후 목록 유지 확인 (DB 저장 여부)

## 9. 실시간 감시 & 이벤트 로그

**목적**: 파일 변경 감지 → 로그 저장 → AI 분석 흐름 확인

### 9.1 감시 시작

1. 메인 대시보드(`/`) → "감시 시작" 버튼 클릭
2. 상태가 "감시 중"으로 변경 확인

### 9.2 파일 이벤트 생성

1. 감시 폴더 내에서 파일 생성:
   ```bash
   echo "test" > /path/to/watched/folder/test.txt
   ```
2. `http://localhost:3000/logs` 접속
3. 이벤트 로그 목록에 `CREATE` 이벤트 표시 확인
4. 파일 수정 후 `MODIFY`, 삭제 후 `DELETE` 이벤트 확인

### 9.3 AI 분석 (AI 서버 실행 중)

1. 3초 이상 빠르게 여러 파일을 수정 (AI 분석 윈도우 트리거)
2. `http://localhost:3000/notifications` 에서 알림 발생 확인
3. 알림 레벨(SAFE / WARNING / DANGER) 확인

## 10. 감시 폴더 삭제 감지

**목적**: 감시 중인 루트 폴더 자체가 삭제될 때 DANGER 즉시 발생

### 단계

1. 테스트용 임시 폴더 생성:
   ```bash
   mkdir /tmp/watch_test
   ```
2. `/tmp/watch_test`를 감시 폴더로 추가
3. 메인 대시보드에서 "감시 시작"
4. **감시 폴더 자체를 삭제**:
   ```bash
   rm -rf /tmp/watch_test
   ```
5. `http://localhost:3000/notifications` 에서 DANGER 알림 발생 확인
6. 알림 상세에서 `aiScore=1.0` 확인

**기대 결과**: AI 서버 없이도 즉시 DANGER 알림 생성 (score=1.0)

## 11. 수동 스캔 + 결과 모달

**목적**: 수동 스캔 시작 → 진행률 표시 → 완료 모달 확인

### 단계

1. 감시 폴더가 1개 이상 추가된 상태 확인
2. 메인 대시보드에서 "수동 스캔" 버튼 클릭

4. 스캔 완료 후 모달 팝업 확인:
   - "스캔 완료" 제목
   - 스캔된 파일 수 / 전체 파일 수 표시
5. 모달 닫기 버튼 또는 배경 클릭으로 닫기 확인

## 12. 소리 알림 ON/OFF

**목적**: 설정에서 소리를 끄면 탐지 시 경보음이 재생되지 않아야 함

### 소리 ON/OFF 설정

1. `http://localhost:3000/settings/notify` 접속
2. "소리 알림" 체크박스 **해제** 후 저장
3. `localStorage`에 설정이 저장됐는지 확인:
   ```js
   // 브라우저 콘솔
   JSON.parse(localStorage.getItem('watchservice.notifySettings'))
   // 예상: { popup: true, sound: false }
   ```
4. 새 알림 발생 → 경보음 **재생되지 않음** 확인

### 소리 ON 확인

1. "소리 알림" 체크박스 **체크** 후 저장
2. 새 알림 발생 → 심각도별 경보음 재생 확인
   - DANGER: 강한 경고음 (880Hz → 660Hz → 880Hz)
   - WARNING: 중간 알림음

## 13. 이메일 알림

**목적**: DANGER 탐지 시 설정된 이메일로 경보 메일 발송 확인

### 13.1 SMTP 환경변수 설정

Gmail 앱 비밀번호 발급 방법:
1. Google 계정 → 보안 → 2단계 인증 활성화
2. 앱 비밀번호 생성 (앱: 메일, 기기: 기타)
3. 생성된 16자리 비밀번호를 `SMTP_PASSWORD`로 설정

```bash
SMTP_HOST=smtp.gmail.com
SMTP_PORT=587
SMTP_USERNAME=your@gmail.com
SMTP_PASSWORD=xxxx xxxx xxxx xxxx   # 띄어쓰기 없이 입력
```

### 13.2 이메일 설정 페이지 테스트

1. `http://localhost:3000/settings/email` 접속
2. 수신 이메일 주소 입력 후 "저장" 클릭
3. "테스트 메일 발송" 버튼 클릭
4. 메시지: "테스트 메일을 xxx@xxx.com 로 발송했습니다. 메일함을 확인해 주세요."
5. 해당 이메일 수신함 확인: "[WatchService] 이메일 알림 테스트" 제목 메일 수신

### 13.3 DANGER 탐지 시 이메일 자동 발송

1. 이메일 주소 저장된 상태에서 감시 폴더 삭제 (섹션 10 방법 사용)
2. DANGER 알림 발생 확인
3. 수신함에서 "[WatchService] DANGER — 랜섬웨어 의심 활동 탐지" 메일 수신 확인
4. 메일 내용 확인:
   - 탐지 시각
   - 위험도 (DANGER, 점수)
   - 패밀리명
   - 영향 파일 수
   - AI 분석 내용
   - 대응 방법

### 13.4 이메일 미설정 시 발송 안 됨

1. 이메일 주소 비워두고 저장
2. DANGER 탐지 발생
3. 백엔드 로그에 `수신 이메일 미설정 — 발송 건너뜀` 확인
4. 이메일 수신 안 됨 확인

## 14. Electron 긴급 전체 화면 경보

**목적**: DANGER 탐지 시 전체 화면 경보창 자동 팝업 확인

> **전제조건**: Electron 앱 실행 중 (`cd watchservice_electron && npm start`)

### 단계

1. Electron 앱이 정상 실행돼 브라우저 창이 열린 상태 확인
2. Electron 창을 최소화
3. 감시 폴더 삭제로 DANGER 탐지 유발 (섹션 10 방법)
4. 3초 이내 전체 화면 경보창 자동 팝업 확인:
   - 붉은 배경 (#dc2626)
   - "DANGER" 텍스트 + 탐지 시각 표시
   - 경보음 3연속 재생 (880Hz → 660Hz → 880Hz)
5. "확인" 버튼 클릭으로 창 닫힘 확인
6. 같은 알림으로 경보창 중복 팝업 **안 됨** 확인 (id 비교 로직)

## 21. Electron 앱 디버그 & 재설치 명령어

> 패키징된 Electron 앱(`.dmg` 설치본)에서 문제가 생겼을 때 사용하는 명령어 모음.

---

### 21.1 터미널에서 직접 실행해 로그 보기

Electron 앱을 GUI로 실행하면 Spring Boot/Electron 메인 프로세스 로그가 안 보인다.  
터미널에서 실행하면 `[Spring]`, `[Electron]` 로그가 콘솔에 바로 출력된다.

```bash
"/Applications/WatchService Agent.app/Contents/MacOS/WatchService Agent"
```

> DevTools 콘솔(`⌘⌥I`)은 렌더러(React) 프로세스 로그만 보인다. Spring Boot 기동 오류는 반드시 터미널 실행으로 확인.

---

### 21.2 포트 8080 프로세스 강제 종료

앱을 트레이 아이콘에서 "숨기기(Hide)"만 하면 Spring Boot가 계속 8080을 점유한다.  
새 인스턴스를 실행하면 `Address already in use: 8080` 오류가 발생한다.

```bash
# 8080 포트를 사용 중인 PID 확인
lsof -i :8080

# 강제 종료 (모든 8080 프로세스 한 번에)
lsof -ti :8080 | xargs kill -9
```

> 앱을 완전히 종료하려면 반드시 **트레이 아이콘 우클릭 → 종료** 를 사용한다.

---

### 21.3 macOS Gatekeeper 격리 해제

DMG에서 설치한 직후 "손상된 앱" 오류가 뜨거나 앱이 열리지 않을 때:

```bash
sudo xattr -cr "/Applications/WatchService Agent.app"
```

> 서명되지 않은 빌드를 설치할 때마다 필요. Apple Silicon(M1/M2/M3)에서 더 자주 발생.

### 21.6 자주 발생하는 Electron 앱 오류

| 증상 | 원인 | 해결 |
|------|------|------|
| "서비스 시작 실패" 팝업 | Spring Boot가 60초 내에 헬스체크를 통과 못 함 | 터미널 실행으로 Spring 로그 확인 |

| 2 | `.txt`, `.pdf`, `.docx` 등 다양한 파일 이벤트 발생 후 새로고침 | 각 확장자 비율 업데이트 |
| 3 | 로그가 0건인 경우 | "데이터 없음" 안내 표시 (차트 에러 아님) |

**API 확인**:
```bash
curl http://localhost:8080/logs/stats/extensions
# [{"ext":"txt","count":12},{"ext":"pdf","count":5},...]
```

### 22-10. 자주 변경되는 파일 차트

**접속 경로**: `로그 → 자주 변경 파일` (`/logs/top-files`)

| 단계 | 행동 | 기대 결과 |
|------|------|----------|
| 1 | 페이지 접속 | 변경 횟수 기준 상위 파일 목록 + 차트 표시 |
| 2 | 특정 파일을 반복 수정 (5회 이상) 후 새로고침 | 해당 파일이 상위에 표시 |
| 3 | Top-N 필터 변경 (5 / 10 / 20) | 표시 개수 변경 확인 |

**API 확인**:
```bash
curl "http://localhost:8080/logs/stats/top-files?limit=10"
# [{"path":"/watched/file.txt","count":8},...]
```

### 22-11. 랜섬웨어 패밀리 상세 정보

**접속 경로**: 알림 상세 페이지 → 패밀리명 클릭

| 단계 | 행동 | 기대 결과 |
|------|------|----------|
| 1 | DANGER/WARNING 알림 상세 접속 | `topFamily` 값 표시 (예: "LockBit") |
| 2 | 패밀리명 클릭 | 상세 모달 팝업 |
| 3 | 모달 내용 확인 | 패밀리명, 첫 발견 시기, 공격 방식, 알려진 확장자, 대응 방법 |
| 4 | X 버튼 또는 배경 클릭 | 모달 닫힘 |
| 5 | 지원 외 패밀리명으로 API 직접 호출 | 404 반환 |

**지원 패밀리 (13종)**: WannaCry, LockBit, REvil, Ryuk, DarkSide, Dharma, Conti, Cerber, Gandcrab, Maze, Phobos, Ragnar, Shade

```bash
# 패밀리 정보 직접 조회
curl http://localhost:8080/api/families/WannaCry

# 미지원 패밀리
curl http://localhost:8080/api/families/Unknown
# 기대: 404
```
