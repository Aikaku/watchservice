# WatchService Agent — 발표 예상 질문 & 답변

---

## AI / 모델

**Q1. XGBoost를 선택한 이유가 뭔가요? 딥러닝은 왜 안 썼나요?**

랜섬웨어 탐지에 쓰이는 피처가 9개 수치 데이터라 딥러닝의 강점인 이미지나 시계열 처리가 필요하지 않습니다.
XGBoost는 적은 데이터로도 빠르게 학습되고, 추론 속도가 매우 빠르며, 피처 중요도 해석도 쉽습니다.
실시간 탐지 시스템에서는 정확도 외에 응답 속도와 해석 가능성도 중요하기 때문에 XGBoost가 더 적합한 선택이었습니다.

---

**Q2. 합성 데이터로 학습했는데 실제 환경에서도 잘 동작하나요?**

합성 데이터의 한계는 인정합니다.
다만 저희 탐지 방식은 파일 내용이 아닌 행동 수치(읽기·쓰기·엔트로피 변화 등)를 사용하기 때문에,
실제 랜섬웨어도 결국 동일한 행동 패턴을 가질 수밖에 없습니다.
실제 랜섬웨어 샘플 데이터를 추가로 확보해 재학습하면 정확도를 더욱 높일 수 있습니다.

---

**Q3. 탐지 정확도는 얼마나 되나요?**

합성 데이터 기준 테스트셋(20%)에서 ROC-AUC와 PR-AUC 모두 높게 나왔습니다.
다만 실제 운영 환경에서의 정확도는 학습 데이터 분포와 차이가 있을 수 있습니다.
오탐(정상인데 경보) 가능성을 줄이기 위해 DANGER 기준을 70% 이상으로 설정했습니다.

---

**Q4. 9개 피처 중 어떤 게 가장 중요한가요?**

학습 후 피처 중요도를 확인한 결과, `file_encrypt_like_count`와 `entropy_diff_mean`이 가장 높은 중요도를 보였습니다.
즉, 엔트로피가 급격히 올라가면서 파일이 많이 바뀌는 패턴이 랜섬웨어 판정에 가장 강한 신호입니다.

---

## 탐지 방식

**Q5. 정상적인 압축이나 대용량 파일 복사를 랜섬웨어로 오탐하지 않나요?**

오탐 가능성은 있습니다. 대용량 파일 복사나 압축 시 엔트로피 변화와 쓰기 횟수가 증가하기 때문입니다.
이를 줄이기 위해 두 가지 방어 장치를 뒀습니다.
첫째, DANGER 기준을 확률 70% 이상으로 높게 설정했습니다.
둘째, 예외 규칙 기능을 제공해 특정 폴더나 확장자를 분석에서 제외할 수 있습니다.
오탐이 발생하면 대시보드에서 "오탐 신고" 버튼으로 기록할 수 있습니다.

---

**Q6. 3초 윈도우가 너무 짧거나 길지 않나요?**

3초는 랜섬웨어의 암호화 버스트 패턴을 포착하기 위한 값입니다.
랜섬웨어는 짧은 시간에 대량의 파일을 처리하기 때문에 3초 안에도 충분한 신호가 쌓입니다.
반대로 너무 길게 잡으면 초기 감지가 늦어질 수 있습니다.
이 값은 `application.yml`의 설정으로 조정 가능합니다.

---

**Q7. 랜섬웨어를 탐지만 하고 차단은 못 하나요?**

맞습니다. 현재는 탐지 후 경보 발송에 집중합니다.
파일 시스템 차단은 OS 커널 수준의 드라이버가 필요해 구현 난이도가 훨씬 높습니다.
저희 목표는 피해 확산 전에 사용자가 인지하고 직접 대응할 수 있도록 돕는 것입니다.

---

## 기술 스택

**Q8. 왜 Spring Boot와 Python AI 서버를 분리했나요?**

XGBoost 같은 ML 라이브러리는 Python 생태계가 훨씬 성숙해 있습니다.
JVM 위에서 Python 모델을 직접 실행하면 통합 복잡도가 높아지고 의존성 관리도 어렵습니다.
REST API로 분리하면 AI 서버를 독립적으로 업데이트하거나 교체할 수 있어 유지보수가 쉽습니다.

---

**Q9. SQLite를 DB로 쓴 이유가 있나요?**

단일 사용자 데스크탑 앱이기 때문입니다.
MySQL이나 PostgreSQL 같은 서버형 DB는 별도 설치가 필요해 사용자 진입 장벽이 높아집니다.
SQLite는 파일 하나(`log.db`)로 동작해 설치 없이 바로 사용할 수 있고, 이 규모의 데이터에서 성능도 충분합니다.

---

**Q10. Gemini API 키가 없으면 어떻게 되나요?**

Gemini API 키가 없으면 대응 가이드 생성 기능이 비활성화됩니다.
이 경우 탐지된 위험도·영향 파일 수·감지된 활동을 바탕으로 자체 생성한 fallback 가이드가 대신 표시됩니다.
탐지와 경보 자체는 Gemini 없이도 완전히 동작합니다.

---

## 시스템 설계

**Q11. 여러 사용자가 동시에 쓸 수 있나요?**

현재는 단일 사용자 데스크탑 앱으로 설계되어 있습니다.
ownerKey를 고정값 `"default"`로 관리하기 때문에 서버 재시작 후에도 설정이 유지되는 장점이 있습니다.
다중 사용자 지원은 ownerKey를 계정 기반으로 변경하면 확장 가능한 구조입니다.

---

**Q12. 앱을 꺼두면 탐지가 안 되지 않나요?**

맞습니다. 앱이 실행 중이어야 탐지가 작동합니다.
다만 창을 닫아도 앱은 종료되지 않고 시스템 트레이로 최소화되어 백그라운드에서 계속 실행됩니다.
Windows는 시작 프로그램 자동 등록, macOS는 로그인 항목 등록을 지원해 재부팅 후에도 자동 시작됩니다.

---

**Q13. 이메일 알림이 안 오면 어떻게 되나요?**

이메일 발송 실패는 경보창 표시와 독립적으로 동작합니다.
전체화면 긴급 경보창은 이메일과 무관하게 항상 표시되기 때문에 이메일이 실패해도 사용자는 즉시 인지할 수 있습니다.
발송 실패 시 서버 로그에 오류가 기록됩니다.

---

## 향후 발전 방향

**Q14. 앞으로 추가할 기능이 있다면?**

세 가지를 생각하고 있습니다.
첫째, 실제 랜섬웨어 샘플 데이터로 모델을 재학습해 정확도를 높이는 것입니다.
둘째, 랜섬웨어 탐지 시 자동으로 네트워크를 차단하거나 의심 프로세스를 격리하는 능동 대응 기능입니다.
셋째, 여러 PC를 중앙에서 모니터링하는 관리자 콘솔로의 확장입니다.

## 엔트로피 / 파일 분석

**Q15. 엔트로피 계산할 때 파일을 전부 읽지 않는다고 했는데, 그러면 정확하지 않은 거 아닌가요?**

파일 전체를 읽으면 대용량 파일에서 성능 문제가 생깁니다.
저희는 파일 크기가 4KB를 넘으면 앞 40%, 중간 40%, 끝 20% 세 구간만 샘플링합니다.
랜섬웨어는 파일 전체를 균일하게 암호화하기 때문에 세 구간 어디를 봐도 엔트로피가 높게 나옵니다.
정상 파일이 일부 구간에서만 엔트로피가 높은 경우와 명확히 구분됩니다.

---

**Q16. 엔트로피가 원래 높은 파일(이미 압축된 파일 등)은 어떻게 처리하나요?**

이미 압축된 ZIP이나 암호화된 파일은 원래부터 엔트로피가 높습니다.
저희는 절댓값이 아닌 **변화량(diff)** 을 봅니다.
압축 파일은 수정 전후 엔트로피 차이가 거의 없기 때문에 오탐이 발생하지 않습니다.
단, 엔트로피 절댓값이 7.2 이상이면서 다른 피처도 의심스러울 때는 추가 가중치를 부여합니다.

---

**Q17. 파일을 읽는 도중 랜섬웨어가 파일을 잠가버리면 어떻게 되나요?**

파일 접근 실패 시 해당 이벤트는 건너뛰고 다음 이벤트를 처리합니다.
파일을 읽을 수 없다는 것 자체도 하나의 신호이며, 다른 파일들의 이벤트가 누적되면서 탐지가 이루어집니다.
랜섬웨어는 보통 수십~수백 개의 파일을 연속 처리하기 때문에 일부 실패가 전체 탐지에 영향을 주지 않습니다.

---

## 실제 테스트 / 신뢰성

**Q18. 실제 랜섬웨어로 테스트해봤나요?**

실제 악성코드를 실행하는 것은 안전 문제로 시도하지 않았습니다.
대신 랜섬웨어의 핵심 행동 패턴인 대량 파일 이름 변경, 고엔트로피 데이터 덮어쓰기를 스크립트로 시뮬레이션했습니다.
이 시뮬레이션에서 DANGER 탐지가 정상적으로 동작하는 것을 확인했습니다.

---

**Q19. 기존 바이러스 백신과 충돌하지 않나요?**

충돌하지 않습니다.
저희 시스템은 파일을 수정하거나 프로세스를 차단하지 않고, 이벤트를 읽고 분석하는 역할만 합니다.
바이러스 백신은 파일 시그니처를 검사하고, WatchService는 행동 패턴을 분석하는 방식이라 상호 보완적으로 동작합니다.

---

**Q20. AI 서버가 다운되면 탐지가 멈추나요?**

AI 서버가 응답하지 않으면 해당 윈도우의 분석은 실패하고 로그에 오류가 기록됩니다.
탐지가 누락될 수 있다는 한계가 있습니다.
개선 방안으로는 AI 서버 헬스체크를 주기적으로 수행해 사용자에게 경고하거나,
오프라인 폴백 모델을 내장하는 방법을 고려할 수 있습니다.

## 운영 환경

**Q21. Windows와 macOS 둘 다 지원하나요?**

네, 지원합니다.
Windows는 MSI 인스톨러, macOS는 DMG 패키지로 배포합니다.
Java NIO WatchService는 OS에 상관없이 동일하게 동작하며,
Electron 역시 크로스 플랫폼을 지원하므로 UI도 동일합니다.

---

**Q22. 감시 폴더를 많이 추가하면 성능에 문제가 없나요?**

Java NIO WatchService는 OS의 파일시스템 이벤트 알림을 사용하기 때문에 폴링 방식보다 훨씬 효율적입니다.
다만 감시 폴더가 매우 많거나 이벤트가 폭발적으로 발생하는 경우 집계 큐에 부하가 생길 수 있습니다.
일반적인 사용자 환경(수십 개 폴더 이내)에서는 CPU와 메모리 사용량이 미미한 수준입니다.

---

**Q23. 네트워크 드라이브나 외장 드라이브도 감시할 수 있나요?**

경로를 직접 입력하면 감시 폴더로 등록할 수 있습니다.
다만 네트워크 드라이브는 연결이 끊기면 감시가 중단되고, 재연결 시 수동으로 다시 시작해야 합니다.
외장 드라이브는 연결된 동안 정상적으로 감시됩니다.

## 개발 과정

**Q24. 개발하면서 가장 어려웠던 점은 무엇인가요?**

크게 두 가지였습니다.

첫째, **이메일 알림 미발송 문제**입니다.
테스트 발송은 되는데 실제 DANGER 탐지 시에는 이메일이 오지 않았습니다.
원인을 추적해보니 두 가지가 복합적으로 작용했습니다.
ownerKey가 서버 재시작마다 바뀌어 이메일 설정을 찾지 못했고,
`@Async` 어노테이션 때문에 예외가 발생해도 조용히 삼켜지는 문제였습니다.
ownerKey를 고정값으로 바꾸고, 비동기 처리를 제거해 해결했습니다.

둘째, **macOS 앱 서명 문제**입니다.
Apple Developer 인증서 없이 배포하면 Gatekeeper가 앱을 차단합니다.
JRE 내부 바이너리들을 개별 서명하고 DMG를 직접 재생성하는 방식으로 우회했습니다.

**Q25. rename 이벤트를 왜 DELETE + CREATE 쌍으로 처리하나요?**

Java NIO WatchService의 특성 때문입니다.
파일 이름이 바뀔 때 WatchService는 rename 이벤트를 별도로 제공하지 않고,
원본 파일의 DELETE와 새 이름의 CREATE 이벤트를 순차적으로 발생시킵니다.
저희는 짧은 시간 안에 발생한 DELETE-CREATE 쌍을 경로·크기 유사도로 매칭해 rename으로 판정합니다.
랜섬웨어는 파일을 암호화한 뒤 `.wncry` 같은 확장자로 rename하기 때문에 이 탐지가 중요합니다.
