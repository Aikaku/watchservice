## 3. `코드/api_server.py` — score 계산 방식 수정

### 기존 상태

`/api/analyze` 엔드포인트에서 위험도 score와 라벨을 다음과 같이 판정했다.

```python
elif "benign" in top_family.lower():
    label = "SAFE"
    score = float(max(0.0, min(1.0, 1.0 - top_prob)))
else:
    score = float(max(0.0, min(1.0, top_prob)))
    label = "DANGER" if score >= 0.70 else "WARNING"
```

top-1 클래스가 benign이면 SAFE, 아니면 WARNING/DANGER로 판정하는 방식이었다.
score는 top-1 클래스의 확률 하나만 사용했다.

### 문제점

다중 클래스 모델(랜섬웨어 패밀리별 분류)에서 오작동한다.

예) `LockBit=0.30, WannaCry=0.28, benign=0.42`인 경우
- 기존: benign이 top-1이므로 → **SAFE**, score=0.58
- 실제: 랜섬웨어 총 확률 0.58 → **WARNING**이 올바른 판정

이진 분류(benign/ransomware)에서는 우연히 동일한 결과가 나오지만, 패밀리 분류 모델로 전환 시 오탐이 발생한다.

또한 `"benign" in top_family.lower()` 문자열 검사는 클래스명에 "benign"이 포함된 다른 이름(예: `ransomware_benign_variant`)이 생기면 오작동할 수 있다.

### 변경 내용

```python
# 라벨 결정
# 다중 클래스에서도 정확하도록 benign 이외 모든 클래스 확률을 합산해 위험도(score)를 계산한다.
# 이진 분류(benign/ransomware)에서는 기존 방식과 동일한 결과를 낸다.
if not top_family:
    label = "UNKNOWN"
    score = None
else:
    ransomware_score = float(sum(
        item.prob for item in items if item.family.lower() != "benign"
    ))
    score = max(0.0, min(1.0, ransomware_score))
    if score >= 0.70:
        label = "DANGER"
    elif score >= 0.50:
        label = "WARNING"
    else:
        label = "SAFE"
```

benign 이외 모든 클래스 확률의 합산값을 score로 사용. 라벨 경계는 ≥0.70 → DANGER, ≥0.50 → WARNING, 미만 → SAFE.
이진 분류에서는 `ransomware_prob = 1 - benign_prob`이므로 기존과 동일한 결과가 나온다.

## 4. `코드/api_server.py` — `/health` 모델 미로드 시 503 반환

### 기존 상태

```python
@app.get("/health")
def health():
    """헬스체크"""
    return {
        "ok": True,
        "model_loaded": model is not None,
        ...
    }
```

서버 시작 시 모델 로드에 실패해도 서버가 계속 기동되었다.
`/health`는 모델 유무와 관계없이 항상 HTTP 200, `"ok": true`를 반환했다.

### 문제점

모델이 없는 상태에서 `/api/analyze`를 호출하면 항상 `label=UNKNOWN`이 반환된다.
백엔드나 모니터링 시스템이 `/health`로 서버 상태를 체크할 때 200을 받으면 정상으로 판단하여, 실제로 탐지가 전혀 동작하지 않는 상태임을 알아채지 못한다.
`"ok": true`이지만 `"model_loaded": false`인 응답 본문을 파싱해야만 문제를 알 수 있는 구조였다.

### 변경 내용

```python
@app.get("/health")
def health(response: Response):
    """헬스체크 — 모델 미로드 시 503 반환"""
    if model is None:
        response.status_code = 503
    return {
        "ok": model is not None,
        "model_loaded": model is not None,
        ...
    }
```

모델이 로드되지 않은 경우 HTTP 503을 반환. `"ok"` 필드도 `true` 고정값에서 `model is not None`으로 변경하여 상태 코드와 응답 본문이 일치하도록 수정.

## 5. `코드/api_server.py` — `fillna(-1)` 중복 제거

### 기존 상태

```python
row, matched, missing = _build_row_from_features(features)
X = pd.DataFrame([row]).fillna(-1)
```

### 문제점

`_build_row_from_features()`는 내부적으로 `features.get(c, -1)`로 누락 피처를 이미 -1로 채워서 반환한다.

```python
row = {c: features.get(c, -1) for c in feature_list}  # 누락은 -1로 채움
```

따라서 이후 `fillna(-1)` 호출은 NaN이 존재하지 않아 아무 효과가 없는 죽은 코드였다.
오히려 코드를 읽는 사람이 "이 시점에 NaN이 발생할 수 있다"고 오해할 여지가 있었다.

### 변경 내용

```python
# 모델 입력 형식으로 변환 (누락 피처는 _build_row_from_features에서 이미 -1로 채워짐)
row, matched, missing = _build_row_from_features(features)
X = pd.DataFrame([row])
```

불필요한 `fillna(-1)` 제거, 주석으로 이유 명시.
